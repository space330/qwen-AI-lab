## HIGH — 安全类

### H1. SSRF bypass via open redirect

- **位置：** `server/tools.js:513`（`web_fetch` 工具）
- **问题：** `web_fetch` 使用 `redirect: "follow"` 发起请求，但 SSRF 守卫（`guardFetchUrl`）仅检查初始 URL。攻击者可构造一个公网 URL（如 `https://evil.com/redirect-to-169.254.169.254`），通过 302 重定向穿透到云元数据服务（AWS/GCP/Azure 的 `169.254.169.254`）或内网服务。
- **证据：**
  ```
  guardFetchUrl("https://evil.com/redirect") → null（放行）
  → fetch 跟随 302 → http://169.254.169.254/latest/meta-data/（内网穿透）
  ```
- **影响：** 云环境下可泄露实例元数据（IAM 凭证、网络拓扑等）
- **修复建议：**
  ```js
  // 方案 A：改为手动跟踪重定向
  response = await fetch(rawUrl, { redirect: "manual", ... });
  if ([301, 302, 303, 307, 308].includes(response.status)) {
    const location = response.headers.get("location");
    if (guardFetchUrl(location)) return { error: "blocked_redirect" };
  }

  // 方案 B：限制最大重定向次数 + 每次检查目标地址
  ```

### H2. SSRF bypass via IPv6-mapped IPv4

- **位置：** `server/tools.js:838-861`（`guardFetchUrl`）
- **问题：** SSRF 守卫用正则匹配 IPv4 私有网段（`/^10\./`、`/^192\.168\./` 等），但 Node.js 的 `URL` 解析器会将 IPv6-mapped IPv4 地址归一化为十六进制格式，导致正则无法匹配。
- **证据（已实测确认）：**
  ```
  guardFetchUrl("http://[::ffff:192.168.1.1]/secret")
    → URL.hostname = "[::ffff:c0a8:101]"
    → 正则 /^192\.168\./ 不匹配
    → 返回 null（放行！）

  guardFetchUrl("http://[::ffff:169.254.169.254]/latest/meta-data/")
    → URL.hostname = "[::ffff:a9fe:a9fe]"
    → 正则 /^169\.254\./ 不匹配
    → 返回 null（放行！）

  guardFetchUrl("http://[::ffff:10.0.0.1]/secret")
    → URL.hostname = "[::ffff:a00:1]"
    → 正则 /^10\./ 不匹配
    → 返回 null（放行！）
  ```
- **影响：** 可绕过 SSRF 守卫访问内网任意服务
- **修复建议：**
  ```js
  // 在 guardFetchUrl 中补充 IPv6-mapped 检测
  if (/^\[::ffff:/i.test(host)) {
    return { url: rawUrl, error: "blocked_host", message: "IPv6-mapped 地址已阻止。" };
  }
  // 或者：解析 hostname 后做 DNS 级别的 IP 检查
  ```

### H3. file_search regex DoS (ReDoS)

- **位置：** `server/tools.js:183-188`（`file_search` 工具）
- **问题：** 用户提供的 `query` 参数直接传入 `new RegExp(query, flags)`，无输入校验。恶意正则（如 `(a+)+$`、`(a|aa)*`）可触发指数级回溯（ReDoS），卡死 agent 工具循环中的一个 step。
- **缓解因素：** 工具有 5s 超时（`timeoutMs: 5000`），不会永久卡死，但会浪费一个 tool step 并产生超时错误。
- **修复建议：**
  ```js
  // 方案 A：限制正则复杂度
  if (args?.regex && query.length > 100) {
    return JSON.stringify({ error: "regex_too_long", message: "正则表达式过长（上限 100 字符）。" });
  }

  // 方案 B：用 try-catch + 超时包裹 RegExp 构造
  // 方案 C：禁用 regex 模式，仅支持纯文本搜索
  ```

## MEDIUM — 功能 / 可靠性

### M1. `withTimeout` 不取消底层操作

- **位置：** `server/agentExecutor.js:169-175`
- **问题：** `Promise.race([promise, timeout])` 超时后只是 reject 了外层 Promise，但底层工具（如 `web_fetch` 的 `fetch()`、`qwenSearch` 的 `callQwen()`）仍在后台运行。对于网络请求类工具，这意味着连接和下载会继续消耗资源。
- **影响：** 资源泄漏（CPU / 内存 / 网络带宽），高并发下可能累积
- **修复建议：**
  ```js
  // 让工具接受 AbortSignal，超时时 abort
  function withTimeout(promise, ms, toolName, signal) {
    // ...
  }
  // 或在各工具的 run() 中使用 AbortController
  ```

### M2. memory store 并发竞态

- **位置：** `server/memoryStore.js:52-81`（`writeMemory`）
- **问题：** `writeMemory` 执行 `load()` → 修改数组 → `save()` 的三步操作，无任何锁机制。两个并发的 `memory_write` 调用可能同时 `load()` 到相同的旧数据，各自追加后 `save()`，后写入的覆盖先写入的。
- **影响：** 极端并发下丢失记忆条目（概率低，因为 `memory_write` 需要人工确认，实际并发几乎为零）
- **修复建议：**
  ```js
  // 简单的串行化队列
  let writeQueue = Promise.resolve();
  export function writeMemory(args) {
    const p = writeQueue.then(() => { /* load + modify + save */ });
    writeQueue = p.catch(() => {});
    return p;
  }
  ```

### M3. 正则 HTML 消毒器边界

- **位置：** `server/responseFormatter.js:266-294`（`sanitizeHtml`）
- **问题：** 后端 HTML 消毒器使用正则逐标签匹配，而非 DOM 解析。虽然白名单策略（`allowedTags` + `allowedAttrs`）在正常情况下有效，但正则解析 HTML 存在固有的边界问题：
  - 嵌套同名标签可能导致非贪婪匹配 `([\s\S]*?)` 提前闭合
  - 属性值中包含 `>` 的边界情况（虽然 `escapeAttr` 已转义输出，但输入侧的解析可能不一致）
- **缓解因素：** 前端 `render.js` 有第二层 DOM-based 消毒（`sanitizeAnswerHtml`），形成双层防御
- **修复建议：** 考虑引入 `jsdom` 或 `happy-dom` 做后端 DOM 解析，或在现有正则基础上增加单元测试覆盖边界 case

### M4. 无速率限制

- **位置：** `server/index.js`（全局）
- **问题：** 所有 API 端点无请求频率限制。虽然服务绑定在 `127.0.0.1`（仅本机访问），但如果未来暴露到局域网或通过反向代理对外，攻击者可无限调用千问 API 消耗额度。
- **修复建议：**
  ```js
  // 简单的内存令牌桶
  const rateLimit = new Map();
  app.use((req, res, next) => {
    const key = req.ip;
    const now = Date.now();
    const bucket = rateLimit.get(key) || { tokens: 30, lastRefill: now };
    // ... 令牌桶逻辑
  });
  ```

## LOW — 边缘 / 代码质量

### L1. `streamingPlainText` 实体解码不完整

- **位置：** `src/components/render.js:610-618`
- **问题：** 流式阶段的纯文本转换只解码了 `&lt;`、`&gt;`、`&amp;` 三种实体，`&quot;`、`&#039;`、`&nbsp;` 等未处理，用户在流式输出阶段会看到原始实体码（如 `&quot;` 而非 `"`）。
- **影响：** 仅影响流式阶段的视觉体验，最终结果（`result` 事件）会替换为完整 HTML
- **修复建议：**
  ```js
  .replace(/&quot;/g, '"')
  .replace(/&#0?39;/g, "'")
  .replace(/&nbsp;/g, " ")
  ```

### L2. `.env` 引号剥离不严谨

- **位置：** `server/config.js:81`
- **问题：** `rawValue.replace(/^["']|["']$/g, "")` 对混合引号（如 `"value'`）会错误剥离首尾引号。实际场景中极少出现，但不符合 `.env` 解析的通用约定。
- **修复建议：**
  ```js
  const value = (rawValue.startsWith('"') && rawValue.endsWith('"'))
    || (rawValue.startsWith("'") && rawValue.endsWith("'"))
    ? rawValue.slice(1, -1)
    : rawValue;
  ```

### L3. 流式路径无重试

- **位置：** `server/qwenClient.js:165-265`（`streamQwenOnce`）
- **问题：** 流式请求（`onDelta` 回调）只执行一次，无重试机制。注释已说明原因："retrying after we have already emitted partial tokens would duplicate output on the client"。
- **影响：** 网络抖动导致的流式中断直接报错，用户需手动重试
- **缓解：** 这是合理的设计权衡，非 bug。建议在前端增加"重试"按钮

### L4. IndexedDB 事务错误可能为 null

- **位置：** `src/data/conversationRepository.js:227-234`
- **问题：** `transaction.onerror` 回调中 `reject(transaction.error)` 的 `error` 可能为 `null`，导致 `reject(null)`，上层 catch 块收到 null 而非 Error 对象，增加调试难度。
- **修复建议：**
  ```js
  transaction.onerror = () => reject(transaction.error || new Error("IndexedDB transaction failed"));
  ```
