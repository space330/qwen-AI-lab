## P0：必须修复

### 1. 非流式 Agent 接口未进入工具调用循环

- 涉及接口：`POST /api/qwen/chat`
- 当前表现：即使请求使用 `mode: "agent"`，仍按照普通 Chat 链路处理。
- 实测结果：要求 Agent 调用计算器时，最终返回的 `toolCalls` 为空。
- 影响：Agent 与 Chat 没有在所有后端接口中真正区分。

修复要求：

- 非流式 Agent 请求也必须执行完整 Agent 工具调用循环。
- 返回最终答案以及完整工具执行步骤。
- Chat 模式继续保持普通模型调用，不允许自动调用工具。

验收要求：

- 向 `/api/qwen/chat` 发送 `mode: "agent"` 的计算任务时，能够真实执行 `calculator`。
- 同样任务使用 `mode: "chat"` 时，不产生工具调用。

### 2. 缺少统一、完整的 `toolSteps` 数据契约

当前表现：

- 前端使用 `steps` 保存和渲染工具步骤。
- 后端最终响应使用 `toolCalls`。
- 流式事件分别使用 `tool_start`、`tool_result` 和 `tool_error`。
- 字段名称和数据结构尚未统一。
- 最终响应中的 `toolCalls` 只保存工具名称和参数，没有完整执行结果。

当前缺少：

- 工具执行状态。
- 工具执行结果。
- 工具错误信息。
- 工具执行耗时。
- 统一的步骤标识和最终状态。

影响：

- 非流式客户端无法获得完整工具执行历史。
- 前端、后端和持久化层需要分别适配不同结构。
- 后续增加重试、人工确认和工具审计时难以扩展。

建议统一结构：

```json
{
  "step": 1,
  "tool": "calculator",
  "status": "completed",
  "args": {
    "expression": "19 * 23"
  },
  "result": "{\"result\":437}",
  "error": null,
  "durationMs": 12
}
```

修复要求：

- 统一使用 `toolSteps` 作为正式字段。
- 流式事件、最终响应、前端状态和 IndexedDB 使用同一结构。
- 最终响应必须包含完整步骤状态、结果、错误和耗时。

## P1：正式发布前修复

### 3. 联网搜索仍使用 Mock 数据

- 当前表现：未配置 `BOCHA_API_KEY` 时，`web_search` 返回本地占位结果。
- 当前环境状态：真实 Bocha 搜索未配置。
- 影响：Agent 无法真正查询实时外部信息。
- 风险：用户可能误将 Mock 结果当作真实搜索结果。

修复要求：

- 配置并验证真实搜索服务。
- Mock 搜索结果必须在前端明确标记。
- 正式环境不应静默降级为 Mock。
- 搜索服务失败时应提示真实失败原因，并允许 Agent 基于已有信息继续回答。

验收要求：

- 使用真实搜索服务查询近期信息，返回可验证的真实结果。
- 搜索失败时不会导致整个 Agent 请求崩溃。

### 4. 工具调用总步骤数缺少严格限制

- 当前表现：系统限制的是模型工具调用循环轮次，而不是实际工具调用总步骤数。
- 风险：模型单轮返回多个工具调用时，总工具步骤数可能超过限制。
- 影响：
  - 请求成本可能失控。
  - 响应时间可能过长。
  - Agent 可能执行过多工具调用。

修复要求：

- 增加 `AGENT_MAX_TOOL_STEPS` 配置。
- 每次执行工具前检查已经执行的总步骤数。
- 达到限制后停止执行后续工具，并要求模型生成最终回答。
- 最终响应中标记是否因为达到步骤限制而收束。

验收要求：

- 模型单轮返回多个工具调用时，总执行数仍不会超过配置上限。
- 达到上限后 Agent 能正常输出最终答案。

### 5. V2.0 版本信息不一致

当前版本信息：

- README：`2.0.0`
- `package.json`：`1.2.0`
- `package-lock.json` 项目版本：`1.2.0`
- 本地数据导出版本：`1.2.0`

影响：

- 无法准确识别当前应用版本。
- 数据导出与未来迁移可能出现版本判断错误。
- 发布包和文档信息不一致。

修复要求：

- 将应用版本统一更新为 `2.0.0`。
- 更新 `package.json` 和 `package-lock.json`。
- 更新本地数据导出版本。
- 检查 README、发布说明和验收文档中的版本号。
